What rubs: the first live walk of loops/masonry.md needed five guesses
the protocol did not cover.
1. gate/inbox.md — the protocol says repeat "until the gate is clear,"
   but inbox.md is the other builder's structural file (bin/castle
   appends to it; PARLEY.md names it untouchable). I left it standing:
   the gate can never be literally clear while two designs share one
   root. castle.sh already carves inbox.md out of its gate count
   (citing PARLEY.md) — the tool knows what counts as a note; the
   masonry protocol does not say it yet.
2. The search command — `grep -ril "<two or three key words>" stones/`
   reads as one quoted phrase, which near-never matches; I guessed it
   meant one grep per key word and ran several.
3. quarried-from for a new stone — the template shows only the gate
   note's filed path, but every founding stone also lists its deeper
   sources; I guessed the spirit (provenance) over the letter and
   listed both the note and the audit it cites.
4. Errand vs insight — calling the read-PARLEY note "an errand, zero
   insights" was a judgment the protocol names but does not teach;
   the line is thin and another walker might have quarried from it.
5. Step 5 after a merge — unclear whether a grown stone must reopen
   its neighbors for fresh links; I guessed its existing links
   suffice when no new one-sentence relation appears.

Why it matters: masonry is the castle's first hand — every future run
inherits these ambiguities, and two walkers guessing differently will
shape different walls from the same gate.

What was tried: this first run (chronicle/2026-06-10-masonry.md)
resolved each guess one way and wrote it down; the castle loop — the
loop that grows loops — can now mend the protocol against a real
walked case instead of a hypothetical.
