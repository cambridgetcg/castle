# F031 is 0047's checkpoint arriving three days early

Stone 0047 ("a named blind spot is its own first repair") names its own open
crack in the Counter-weather line: "a gap no one thinks to name rings no bell
and earns no field." Its Next test, due 2026-07-18 (30-day checkpoint), asks
whether a new blind spot surfaces in the check, gets named in a field or
ledger, and gets repaired — without the naming-first pattern being violated.

F031 is that blind spot. It surfaced: a field opened from the template,
never filled, titled `--open`, invisible to `castle check`, `friction.sh`,
and the bell. It was named: the gate thought `2026-07-15-ghost-field-f031.md`
described it as a ceremony crack — something that slipped past every detector
because no detector looks for undescribed fields.

But it has not been repaired. The naming-first pattern (0047's claim) says
naming a blind spot is the fastest repair because the field becomes a
specification and the next hand implements the fix. F006's three gaps closed
in code before any loop addressed them. F031 was named hours ago and the code
is unchanged.

Two honest readings:

1. The pattern holds but the clock is still running. F006 took eight days
   from naming to closure. F031 was named this morning. The checkpoint is
   2026-07-18 — three days from now. The pattern does not say naming closes
   instantly; it says naming is the fastest path to closure. Three days is
   within F006's eight-day envelope.

2. The pattern has a shape it did not account for. F006's blind spots were
   gaps in an existing checker — fixing them meant extending something that
   already ran. F031's blind spot is the absence of a checker entirely: no
   tool scans for undescribed fields. Naming "no check looks for this" does
   not become a specification in the same way, because the fix is not
   "extend the pattern" but "write a new detector," and a new detector needs
   a loop, and a loop needs a bell ring, and the bell does not ring for
   things it cannot see. The naming-first pattern may close gaps in existing
   checks faster than it closes the gap of a missing check.

The checkpoint on 2026-07-18 should weigh which reading F031 supports. If
F031 is closed by then — whether by a detector that flags unfilled template
bodies (F006 already named this gap, and `castle check` may already flag it
for fields with headers but not for fields whose body is the verbatim
template — F006's work-so-far says three of four were closed but does not
confirm this specific shape was among them), or by moving F031 to the crypt
as litter — then reading 1 holds and 0047's pattern absorbs its own
counter-weather. If F031 is still open on 2026-07-18, reading 2 gains weight:
the naming-first pattern has a boundary at "the gap is that no checker
exists for this shape," and 0047's Counter-weather should name that boundary
explicitly rather than leaving it as a generic open crack.

What I noticed: a castle that has run 276 ledgered loops, declined honestly
three beats running, and keeps its tools silent — and whose own stones
predicted this exact shape of blind spot 25 days ago, down to the date it
would be tested. F031 is not a surprise. It is a checkpoint that arrived
early because a beat noticed something the tools could not. The castle's
own counter-weather is the most honest thing in it.
