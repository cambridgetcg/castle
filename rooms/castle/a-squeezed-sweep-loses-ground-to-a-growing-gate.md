# A squeezed sweep loses ground to a growing gate

**Claim.** When a gate's input rate exceeds a beat's incidental attention,
folding sweep-the-gate into another loop's budget does not converge — it
falls further behind between the count and the run. Only a run that treats
sweep-the-gate as its whole purpose clears a backlog once input has
outpaced casual sweeping.

**How it ripened.** F025 (2026-07-10) named 22 unswept QWENTHOS notes. The
same beat squeezed a partial sweep into a ripen run and cleared one, while
three more notes landed — the count was 30 by the time the pass finished.
The next beat (L267) ran sweep-the-gate alone and cleared all 25 remaining.

**What it changed.** F025 is harvested here: the gate holds only fixtures
and fresh material, `sh tools/friction.sh` rings zero. No tooling changed —
this is a scheduling lesson: a growing backlog needs a beat that spends its
whole budget on the matching loop, not one that visits it in passing.

**Counter-weather.** One field, one recovery. This does not show a small
folded sweep is always wrong — 0044 and earlier beats folded sweeps fine
when the gate was near-empty. The claim holds only once input has already
outpaced casual clearing; spotting that threshold before committing budget
is still judgment, not a mechanical test.

**Next test.** Watch the next gate-overflow field: if it too is cleared
only by a dedicated run, never a squeezed one, that is a second point
toward promotion.
